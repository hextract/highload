# Точки роста PoC (baseline → итерации)

Baseline намеренно оставляет узкие места, которые можно снять измеримыми правками в `iter-1`, `iter-2` и далее.

## Что упрощено в baseline

- **Поиск ресторанов** — только PostgreSQL, без Elasticsearch и без Redis cache-aside для меню (в отличие от целевой архитектуры).
- **Межсервисный контракт** — HTTP JSON вместо gRPC между gateway и сервисами; Kong не используется, стоит nginx.
- **События оплаты** — публикация в Kafka сразу после `COMMIT`, без transactional outbox и без гарантии «at-least-once» при падении между коммитом и `WriteMessages`.
- **Индексы** — минимальный набор (в основном PK/FK); нет целевых индексов под `orders(user_id, created_at)`, гео/кухню для hot paths.
- **Пулы и лимиты** — дефолтные настройки `pgxpool` и брокера; нет явного тюнинга под 2 vCPU / 8 GB VM.
- **Потребление Kafka** — одна consumer group, без DLQ и без строгой идемпотентности на стороне payment при повторной доставке сообщения (частично компенсируется `UPDATE ... WHERE status = 'pending'`).

## Iteration 1 (типичный фокус)

- **Индексы и запросы**: `EXPLAIN ANALYZE` на `GET /restaurants`, загрузку меню, создание заказа; добавить индексы под фильтры (`cuisine`, `is_active`), при необходимости — выражение для расстояния или ограничить выборку bbox-ом до Haversine.
- **Пулы**: настроить `MaxConns` / `MinConns` / `MaxConnLifetime` в `pgxpool` под лимиты контейнера и число воркеров k6.
- **N+1 / round-trips**: при росте меню — один запрос с агрегацией JSON или жёсткая нормализация; убрать лишние `SELECT` в горячих путях.
- **Redis (cache-aside)** для `GET .../menu` с TTL 5 мин (как в [architecture.md](architecture.md)) — снижение нагрузки на HDD Postgres под read-heavy профилем.

## Iteration 2 (события и устойчивость под нагрузку)

- **Transactional outbox**: запись события в таблицу outbox в той же транзакции, что и заказ/платёж; отдельный процесс/горутина публикует в Kafka — устраняет разрыв commit vs produce.
- **Circuit breaker / лимиты** на HTTP-вызовы catalog из order (сейчас только таймаут клиента); деградация при падении каталога.
- **Kafka**: размер батчей, компрессия, `linger`, отдельные топики/partitions под write path; при необходимости — увеличение ресурсов redpanda в рамках VM.
- **Горизонтальное масштабирование** stateless `order-service` за nginx + `docker-compose.scaled.yml` (несколько реплик, sticky не обязателен при stateless API).
- **Долгий хвост**: опционально read replica Postgres для read-only эндпоинтов (поиск/трекинг) — отдельный DSN и маршрутизация в коде.

## Дальше (вне минимальных двух итераций)

- Elasticsearch для полнотекста и геопоиска под целевые 2 800 RPS каталога.
- Разделение БД по сервисам (отдельный инстанс/схема на сервис) и контракты версионирования событий.
- Полноценный Kong, JWT, rate limiting per user, WebSocket трекинг.
