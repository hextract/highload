# ADR-002: PostgreSQL как основное хранилище + специализированные БД

## Контекст

Системе нужно хранить данные с разными паттернами доступа:
- **Заказы и платежи** - строгая консистентность, ACID-транзакции, ~900K записей/день, рост до 1.6 TB/год
- **Каталог ресторанов и меню** - преимущественно чтение, ~50K ресторанов, 1.5M позиций меню. Редкие обновления от ресторанов
- **Поиск** - полнотекстовый по названиям, фильтрация по кухне/рейтингу, геопоиск по расстоянию. 2 800 RPS в пике
- **Трекинг и сессии** - горячие данные с коротким TTL: координаты курьеров, статусы активных заказов. Sub-millisecond latency
- **Лог событий** - append-only поток событий, ~6.75 GB/день, нужен replay и fan-out на несколько потребителей

## Рассмотренные альтернативы

### Вариант A: Единый PostgreSQL для всего

**Плюсы:** одна БД, одна экспертиза, нативные транзакции для всего.

**Минусы:** PostgreSQL full-text search заметно медленнее Elasticsearch на 50K документов с геофильтрацией. Хранить координаты курьеров с обновлением каждые 5 секунд в PG - лишняя запись на диск. При 2 800 RPS поисковых запросов PG без кэша не удержит SLO p50 < 150 ms.

### Вариант B: MongoDB как основная БД

**Плюсы:** гибкая схема, встроенный шардинг.

**Минусы:** нет ACID-транзакций между коллекциями в том же виде, что PG. Для платёжного контура, где расхождение < 1 на 100M транзакций (НФТ-006), это критично.

### Вариант C: PostgreSQL + Redis + Elasticsearch + Kafka

**Плюсы:** каждое хранилище под свой паттерн. PG - ACID для заказов/платежей. Redis - sub-ms для кэша и горячих данных. Elasticsearch - полнотекст + гео для каталога. Kafka - durable event log с replay.

**Минусы:** 4 системы хранения - больше операционной нагрузки. Нужна синхронизация PG → ES для каталога.

## Решение

Принят **вариант C: polyglot persistence**.

| Хранилище | Назначение | Обоснование |
|-----------|-----------|-------------|
| **PostgreSQL** | Заказы, платежи, рестораны, пользователи | ACID для финансовых данных, зрелая репликация, JSONB для гибких полей |
| **Redis** | Кэш каталога, сессии, координаты курьеров, rate limiting | Sub-ms latency, TTL для горячих данных, Geo-commands для координат |
| **Elasticsearch** | Полнотекстовый поиск ресторанов, фильтрация, геопоиск | Inverted index + geo_point, горизонтальное масштабирование читающей нагрузки |
| **Kafka** | Шина событий, event sourcing log | Durable append-only log, replay, fan-out, exactly-once semantics |

## Последствия

**Положительные:**
- Платёжный контур на PG с синхронной репликацией - durability < 1 потеря на 10M
- Каталог в ES держит 2 800 RPS поиска с p50 < 50 ms (индекс в RAM)
- Redis кэширует меню ресторанов - снимает нагрузку с PG, p50 < 5 ms
- Kafka гарантирует, что события заказа не теряются при падении consumer-ов

**Отрицательные:**
- PG → ES синхронизация через CDC (Debezium) или двойную запись - eventual consistency каталога (задержка ~1–3 с)
- 4 системы хранения требуют мониторинга, бэкапов, обновлений для каждой
- Команде нужна экспертиза по всем четырём технологиям
