# Требования к системе

## 1. Функциональные требования

1. **Поиск и каталог ресторанов**. Пользователь может найти ресторан по названию, кухне, рейтингу, средней стоимости заказа и расстоянию от текущей локации.

2. **Просмотр меню и конфигурирование блюд**. Пользователь может просматривать меню выбранного ресторана с категориями, фото и описаниями блюд, ценами. Может добавить позиции в корзину.

3. **Управление корзиной и оформление заказа**. Пользователь может управлять корзиной: добавлять/удалять позиции, менять количество, выбирать адрес, способ и время доставки. После он может создать заказ и перейти к оплате.

4. **Онлайн-оплата и обработка платёжных статусов**. Пользователь может оплатить заказ онлайн. Система инициирует платёж, отслеживает статусы и обеспечивает согласованность между состоянием заказа и платёжной транзакцией.

5. **Отслеживание статуса доставки в реальном времени**. Пользователь видит текущий статус заказа и примерное время доставки.

6. **Уведомления**. Система отправляет пользователю уведомления о ключевых событиях.Основной канал - push‑уведомления, резервный для критичных событий - SMS/e‑mail.

## 2. Нефункциональные требования

### 2.1. Performance: Latency

1. **Оформление заказа и онлайн-оплата**  
   Сценарий: от нажатия «Оформить и оплатить» до экрана «Заказ принят / оплата обрабатывается».
   - SLI: end-to-end response time (p50, p99) для `POST /api/orders/checkout`.
   - Условия нагрузки: до 550 RPS по этому эндпоинту в пике (см. раздел 3.2).
   - SLO: p50 < 400 ms при 550 RPS; p99 < 2 000 ms при 550 RPS.

2. **Поиск ресторанов и просмотр каталога**
   - SLI: response time (p50, p99) для `GET /api/restaurants/search` и `GET /api/restaurants/{id}/menu`.
   - Условия нагрузки: до 2 800 RPS по этим эндпоинтам в пике.
   - SLO: p50 < 150 ms при 2 800 RPS; p99 < 500 ms при 2 800 RPS.

3. **Обновление статуса доставки и трекинг**
   - SLI: response time (p50, p99) для `GET /api/orders/{id}/status`.
   - Условия нагрузки: до 2 200 RPS по запросам статусов/обновлениям.
   - SLO: p50 < 100 ms при 2 200 RPS; p99 < 300 ms при 2 200 RPS; лаг между фактической сменой статуса (у курьера/ресторана) и отображением у клиента < 3 секунд в 99 % случаев.

### 2.2. Performance: Throughput

4. **Пропускная способность API**
   - SLI: суммарный RPS по внешнему API (gateway).
   - SLO: система выдерживает не менее 6 000 RPS суммарно при выполнении latency SLO выше (пиковая оценка = 5 500 RPS с запасом на рост).

### 2.3. Availability

5. **Доступность компонентов**  
   Используется связь SLA ↔ downtime (например, 99.99 % = 53 минуты простоя в год).
   - Платёжный контур: availability > 99.99 % (SLI - процент успешных запросов в месяц).
   - Создание/изменение заказов: > 99.95 %.
   - Каталог и поиск: > 99.9 %.
   - Уведомления: > 99.5 %.

### 2.4. Durability

6. **Сохранность заказов и платежей**
   - Заказы: вероятность потери подтверждённого заказа (пользователь увидел «заказ принят») < 1 на 10 000 000 успешных.
   - Платежи: вероятность расхождения состояния между системой и платёжным провайдером < 1 на 100 000 000 транзакций.  
   Практическое следствие: запись заказа и платёжного события в надёжное хранилище до ответа клиенту.

### 2.5. Scalability

7. **Масштабируемость**
   - Поддержать рост с 12 до 20 млн MAU за 3 года без радикального переписывания архитектуры (горизонтальное масштабирование допустимо).
   - При увеличении трафика в 3 раза (до ~18 000 RPS) достаточно пропорционально масштабировать stateless‑сервисы и кластеры БД/кэшей, рост p99 - не более 30 %.

## 3. Оценка нагрузки (Capacity Estimation)

### 3.1. MAU -> DAU -> одновременный онлайн

**Шаг 1. MAU -> DAU**

- MAU: 12 млн (допущение).
- Типичное соотношение DAU/MAU: 10–30 %; берём 25 % для запаса.

**DAU** = 12 000 000 * 0,25 = 3 000 000

**Шаг 2. Запросы на пользователя**

- В среднем 10 запросов к backend на активного пользователя в день (поиск, меню, статусы, checkout).
- Req/day = 3 000 000 * 10 = 30 000 000.

**Шаг 3. Средний RPS**

- 1 день = 100 000 секунд.  
  Avg RPS = 30 000 000 / 100 000 = **300 RPS**.

**Шаг 4. Пиковый RPS**

**Peak RPS** = (DAU * Req/user * Peak_coef) / (Peak_hours * 3600)

Допущения: обеденный пик 3 часа (11:00–14:00), Peak_coef = 2.

**Peak RPS** = (3 000 000 * 10 * 2) / (3 * 3600) = 60 000 000 / 10 800 = 5 555 RPS

Округляем до **= 5 500–5 600 RPS**. Итого: средний RPS = 300, пиковый = 5 500.

**Шаг 5. Одновременный онлайн**

- Активный пользователь в пике - в среднем 1 запрос раз в 5 секунд (0,2 RPS на пользователя).
- N_concurrent = 5 555 / 0,2 = 27 800 -> порядка **25–30 тыс.** одновременно активных пользователей в пике.

### 3.2. RPS по операциям

Разбивка пика = 5 500 RPS по типам (каталог 50 %, трекинг 40 %, checkout 10 %):

- Каталог/поиск: 5 555 * 0,5 = **2 780 RPS** (в НФТ - 2 800).
- Трекинг статуса: 5 555 * 0,4 = **2 220 RPS** (в НФТ - 2 200).
- Checkout: 5 555 * 0,1 = **555 RPS** (в НФТ - 550).

**Число заказов**

- 30 % DAU в день делают хотя бы один заказ -> Orders/day = 900 000.
- Средний TPS по заказам: 900 000 / 100 000 = 9 заказов/сек; с учётом пика 2* - = 18–20 заказов/сек.  
  (550 RPS по checkout включают повторы, неуспешные попытки и сопутствующие запросы.)

### 3.3. Latency budget для checkout

Целевой SLO (НФТ‑001): end-to-end p99 < 2 000 ms при ~550 RPS.

Оценка по цепочке: клиент -> API Gateway -> Auth/Session -> Order Service -> Inventory/Restaurant -> Payment Gateway (external) -> Order Service (финализация) -> ответ.

| Компонент | p99 (ms) |
|-----------|----------|
| API Gateway (роутинг, валидация, rate limiting) | 30 |
| Auth/Session (токен, профиль в Redis/БД) | 20 |
| Order Service (корзина, черновик заказа) | 80 |
| Inventory/Restaurant (доступность, слоты кухни) | 80 |
| Payment Gateway (RTT + 3‑D Secure) | 700 |
| Order Service (фиксация, запись в БД) | 50 |
| Сеть и очереди между сервисами | 20 |
| **Сумма** | **980** |

Буфер на очереди, GC и хвостовые перцентили = 600 ms -> server-side p99 = 1 600 ms. Клиентская сеть ~300–400 ms -> общий UX < 2 000 ms в p99. Бутылочное горлышко - платёжный провайдер (~700 ms).

### 3.4. Оценка хранилища

**Заказы**

- ~5 KB на заказ (хедер + позиции + индексы). В день: 900 000 * 5 KB = 4,5 GB. В год: = 1,64 TB; за 3 года = 4,9 TB. С репликацией (*2–3): горячее хранилище 1 год ~3–5 TB, 3 года ~10–15 TB.

**Журнал событий (Event log)**

- ~15 событий на заказ по ~0,5 KB -> 7,5 KB на заказ. В день: 6,75 GB; в год = 2,46 TB; за 3 года = 7,4 TB. Горячий горизонт 6–12 месяцев - в кластере (Kafka/DWH), старые данные - в холодном object storage.

**Итого:** 1 год - порядка 5–7 TB сырых данных, ~10–15 TB с репликацией; 3 года - до ~50 TB с репликами и запасом.

### 3.5. Bandwidth

Только API (без CDN). Пиковые RPS из п. 3.2.

- Каталог/поиск: 2 780 RPS * 100 KB = 272 MB/s = 2,2 Gbps.
- Трекинг: 2 220 RPS * 2 KB = 4,3 MB/s = 34 Mbps.
- Checkout: 555 RPS * 50 KB = 27 MB/s = 217 Mbps.

Суммарно по API = 304 MB/s = 2,4 Gbps исходящего; с входящим - порядка **= 3 Gbps** в пике. Внутри DC нужен уровень сети не ниже 10 Gbps, медиа - через CDN.

## 4. Допущения, ограничения, открытые вопросы

### 4.1. Допущения

1. MAU на горизонте года - 12 млн; архитектура должна выдерживать рост до 20 млн без переписывания.
2. DAU = 25 % MAU (3 млн).
3. 10 запросов к backend на активного пользователя в день.
4. 30 % DAU в день оформляют хотя бы один заказ.
5. Обеденный пик 3 часа, Peak_coef = 2.
6. Логи и аналитика - в отдельном хранилище (DWH).

### 4.2. Ограничения

1. Ограниченный бюджет - один регион, разумный multi-AZ, без дорогого актив‑актив multi-region.
2. Небольшая команда backend
3. Внешние платёжные провайдеры и СБП - собственные SLA и latency не под нашим контролем.
4. Регуляторика: ФЗ‑152, PCI DSS, требования ЦБ по платежам и персональным данным.
5. Жёсткий time-to-market - часть НФТ (надёжность уведомлений, персонализация) на следующие фазы.

### 4.3. Открытые вопросы

1. Целевой SLA по оплатам и цена простоя для обоснования резервных DC и дублирования провайдеров.
2. Допустим ли сценарий «заказ принят, оплата обрабатывается» с асинхронным доведением результата до пользователя?
3. Обязательные платёжные методы (карта, СБП, кошельки, оплата курьеру) и приоритеты.
4. Нужен ли web‑клиент на первом этапе или достаточно мобильных приложений?
5. Требуемая точность ETA и трекинга (частота обновлений, допустимая погрешность).
6. Планируются ли маркетинговые акции с пиками 10* и выше к обычному обеду?
7. Допустимая деградация при перегрузке: какие функции можно временно отключать (рекомендации, отзывы, сложные фильтры)?

### 4.4. Противоречия и компромиссы

1. **«Оплата не должна теряться» vs бюджет**  
   Абсолютная гарантия тянет за собой геораспределение и дублирование провайдеров. Компромисс: жёсткая durability (НФТ‑006), запись событий заказа/платежа до ответа клиенту, регламент ручной сверки для редких спорных кейсов.

2. **«Быстро и не падать» vs бюджет и команда**  
   SLA 100 % недостижим; компромисс - разные SLA по компонентам (оплата/заказы/каталог/уведомления), на первом этапе простая архитектура с запасом по масштабированию.

3. **«Реальное время» трекинга vs нагрузка и батарея**  
   Частый polling от десятков тысяч клиентов даёт лишнюю нагрузку. Решение: push (WebSocket/long‑poll) с обновлением раз в 3–5 секунд, fallback на редкий polling или SMS для слабых устройств/сетей.

4. **Жёсткий latency везде vs стоимость**  
   Нет смысла держать p99 < 200 ms для второстепенных сценариев. Решение: жёсткие latency SLO только для checkout, каталога и статусов; для остального - более мягкие требования.
